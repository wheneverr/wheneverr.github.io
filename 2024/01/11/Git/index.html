Git是目前世界上最先进的分布式版本控制软件，它是一个源码管理系统，便于多人协作。

版本控制

集中式与分布式

集中式（svn）：有一个单一集中管理的服务器，保存所有文件的修订版本，而协同工作的人们都通过客户端连到这台服务器。
svn每次存到都是差异，需要的硬盘空间会相对小一点，可是回滚的速度会很慢。

  • 优点：代码存放在单一的服务器上，便于项目的管理
  • 缺点：服务器宕机：员工写的代码得不到保障

服务器挂了：整个项目历史记录都会丢失
分布式（git）：客户并不只提取最新版本的文件快照，而是把代码仓库完整地镜像下来。任何一处协同工作的服务器发生故障，时候可以用任何一个镜像的本地仓库恢复。即每次提取都是对代码仓库的完整备份。
git每次存的都是项目的完整快照，需要的硬盘空间会相对大一点，但是通过Git团队的极致压缩，比svn大不了多少，但是回滚速度极快。

安装与初始化

安装

下载安装包，点击下一步。
验证安装成功：
git –version

git config –global user.name “xxx”
git config –global user.email xxx@example.com
git config –list

Git的底层概念（底层命令）

区域

工作区—>暂存区—>版本库

  • 工作区：项目下的所有文件，代码。
  • 暂存区：工作区修改后进入暂存区。
  • 版本库：暂存区累计到一定程度可以生成一个新版本。

初始化新仓库

命令：git init

要对现有的某个项目开始用git管理，只需到此项目所在的目录，执行该命令。
初始化后，在当前目录下会出现一个名为.git的目录，所有git需要的数据和资料都存放在这个目录中。

git目录

hooks 目录包含客户端或服务端的钩子脚本，以后在提交代码之前做的东西放在钩子里面做。钩子相当回调函数，保证提交的东西符合规范。
info 包含一个全局性排除文件，哪些文件git不需要管理的放在info里面。
logs 保存日志信息。
objects 目录存储所有数据内容，相当于数据库。
refs 目录存储指向数据（分支）的提交对象的指针。
config 文件包含项目特有的配置选项。
description 用来显示对仓库的描述信息。
HEAD 文件指示目前被检出的分支。
index 文件保存暂存区信息。

对象

Git对象

git的核心部分是一个简单的键值对数据库。你可以向改数据库插入任意类型的内容，它会返回一个键值，通过该键值可以在任意时刻再次检索该内容。
key:val 组成的键值对（key是val 对应的hash)
键值对在git内部的一个blob类型。不管是存入一个文件还是控制台内容，存入git仓库的都是一个git对象，即都是blob类型。

  1. 向数据库写入内容，并返回对应键值命令：
    1. echo “test content” | git hash-object -w -stdin

-w 指示hash-object命令存储数据对象；若不指定此选项，则该命令仅返回对应的键值
-stdin 指示该命令从标准输入读取内容；若不指定此选项，则须在命令尾部给出待存储文件的路径
test content就是写入文件的内容

  1. git hash-object -w 文件路径
    存文件
  2. git hash-object 文件路径
    返回文件对应键值
  3. 查看Git是如何存储数据的命令：find .git/objects -type fgit存储内容的方式：一个文件对应一条内容。校验和的前两个字符用于命名子目录，余下的38个字符则用作文件名。
  4. 根据键值拉取数据

命令：git cat-file -p d6704….(校验和)
-p指示该命令自动判断内容的类型，并为我们显示格式友好的内容。
返回：test content

  1. 对一个文件进行简单的版本控制
    1. 创建一个新文件并将其内容存入数据库中
      echo “version 1” > test.txt
      控制台输出内容”version 1“写入文件test.txt。生成了一个文件，但还没交给git管理
      git hash-object -w ./test.txt
      保存文件
    2. 向文件里面写入新内容，并再次将其写入数据库：
      vim test.txt
      向其中添加一行内容 version 2 ，wq保存，此时只修改了文件内容，并未提交存入数据库。
      git hash-object -w test.txt
    3. 查看数据库内容
      find .git/objects -type f
      列出objects文件夹中的所有对象，包括git对象，树对象
      git cat-file -p xxx…

问题：

  • git对象只能存储，不能代表一个版本。
  • 记住文件的每一个版本对应的hash值并不现实
  • 在git中，我们只保存了文件内容，并没有保存文件名

解决方案：树对象
注意：当前的操作都是在对本地数据库进行操作，不涉及暂存区。

树对象

树对象（tree object)，它能解决文件名保存的问题，也允许我们将多个文件组织到一起。Git以一种类似于UNIX文件系统的方式存储内容。所有内容均以树对象和数据对象（git对象）的形式存储，其中树对象对应了UNIX中的目录项，数据对象（git对象）则大致上对应文件内容。一个树对象包含了一条或多条记录，一个树对象也可以包含另一个对象。

  1. 查看暂存区当前的样子
    git ls-files -s
  2. 查看树对象
    git cat-file -p master^{tree}（或者是树对象的hash）
    master^{tree}语法表示master分支上最新的提交所指向的树对象。
  3. 构建树对象
    1. 利用update-index命令为test.txt文件的首个版本创建一个暂存区。并通过write-tree命令生成树对象。git update-index –add –cacheinfo 10064\ 83baae…xx test.txt
      git write-tree
      文件模式为 100644，表明这是一个普通文件
      100755，表示一个可执行文件
      120000，表示一个符号链接

–add 因为此前该文件并不存在暂存区中，首次需要add
–cacheinfo 因为要添加的文件位于Git数据库中，而不是位于当前目录下

  1. 新增new.txt，将new.txt和test.txt文件的第二个版本塞入暂存区。并通过write-tree命令生成树对象。echo “new file v1” > new.txt
    git update-index –cacheinfo 10064\ 1f7a7a…xx test.txt
    将test.txt文件的第二个版本放入暂存区：
    git update-index –add new.txt
    会直接执行生成new.txt的git对象，然后放入暂存区，合并了两步，属于高层命令。
    git wtrite-tree
  2. 将第一个树对象加入第二个树对象，使其成为新的树对象
    git read-tree -prefix=bak d8329f…xx(第一个树对象的hash)
    read-tree命令，把树对象读入暂存区。
    git write-tree

解析树对象
Git 根据某一时刻暂存区（即 index 区域）所表示的状态创建并记录 一个对应的树对象，如此重复便可依次记录（某个时间段内）一系列的树对象。
其实树对象是对暂存区内操作的抽象，这颗树对象相对于就是快照。当我 们的工作区有任何更改同步到暂存区时。便会调用 write-tree 命令
通过 write-tree 命令向暂存区内容写入一个树对象。它会根据当前 暂存区状态自动创建一个新的树对象。即每一次同步都产生一颗树对象。且该命令会返回一个 hash 指向树对象。
在 Git 中每一个文件（数据）都对应一个 hash（类型 blob） 每一个树对象都对应一个 hash（类型 tree）
_总结_：我们可以认为树对象就是我们项目的快照

提交对象

我们可以通过调用 commit-tree 命令创建一个提交对象，为此需要指定一个树对象的hash值，以及该提交的父提交对象（如果有的话 第一次将暂存区做快照就没有父对象）

  1. 创建提交对象
    echo ‘first commit’ | git commit-tree d8329f
    返回： fdf4fc3344e67ab068f836878b6c4951e3b15f3dd8329f是第一棵树的树对象，结果就是提交后的对象
  2. 查看提交对象
    git cat-file -p fdf4fc3
    返回： tree d8329fc1cc938780ffdd9f94e0d364e0ea74f579

提交对象的格式 提交对象的格式很简单：
它先指定一个顶层树对象，代表当前项目快照；然后是作者/提交者信息（依 据你的user.name和 user.email 配置来设定，外加一个时间戳）；留空 一行，最后是提交注释

  1. 接着，我们将创建另两个提交对象，它们分别引用各自的上一个提交（作为其父提交对象）：
    echo ‘second commit’ | git commit-tree 0155eb -p fdf4fc3 cac0cab538b970a37ea1e769cbbde608743bc96d0155eb
    是第二次提交的对象，即为第二棵树的树对象，fdf4fc3是第一次提交成功后生成的提交对象，-p说明第一次是第二次的父对象
    echo ‘third commit’ | git commit-tree 3c4e9c -p cac0cab 1a410efbd13591db07496601ebc7a059dd55cfe93c4e9c
    是第三棵树的树对象，cac0cab是第二次提交生成的提交对象


_总结_：本质上项目的版本就是一个提交对象，项目的快照就是一个树对象。

Git的本地操作（高层命令）

git操作最基本的流程

创建工作目录，对工作目录进行修改

  1. git add 工作目录
    这条命令实际上执行了很多条底层命令，包括：git hash-object -w 文件名（修改了多少个工作目录中的文件，此命令就被执行多少次，就会生成多少个git对象）git update-index … （放入暂存区）
  2. git commit -m “注释内容”
    这条命令实际上也执行了很多条底层命令，包括：git write-tree（生成树对象）git commit-tree … （生成提交对象）

git高层命令

git init 初始化仓库
git add . 将修改添加到暂存区
git status 查看当前文件状态（已跟踪或未跟踪）
git commit 直接执行此命令会进入编辑模式进行提交
git commit -m “注释内容” 将暂存区代码提交到仓库
git commit -a 执行此命令git会自动将文件添加到暂存区并提交 （会进入编辑模式）
git diff 查看做了哪些更新还没有暂存
git diff –cached / git diff –staged 查看做了哪些更新已经暂存还未提交
git log 普通的平铺展示提交的历史记录
git log –oneline 极简版展示提交的历史记录

具体使用

在编写代码前，应该先git pull拉取远程仓库中的代码，然后npm i下载新的依赖。
如果要拉取某个分支的代码，则 git pull origin 分支名称
在需要提交新的代码时，先git add 文件相对位置，将新修改的文件加入暂存区，然后git status查看文件的状态，然后git commit -m 注释（修改文件名：相应操作）提交到本地仓库，最后git push提交到远程仓库。
新建分支是git branch 分支名
合并分支是git rebase 要合并的分支名

建立本地仓库

再来复习一下创建新仓库的指令：

git init //把这个目录变成Git可以管理的仓库
git add [README.md](http://README.md) //文件添加到仓库
git add . //不但可以跟单一文件，还可以跟通配符，更可以跟目录。一个点就把当前目录下所有未追踪的文件全部add了 
git commit -m "first commit" //把文件提交到仓库
git remote add origin [git@github.com](mailto:git@github.com):wangjiax9/practice.git //关联远程仓库
git push -u origin master //把本地库的所有内容推送到远程库上

关联github仓库

到github wyj_first仓库复制仓库地址

然后执行指令： git remote add origin git@github.com:gain-wyj/wyj_first.git

如果你clone下来一个别人的仓库，在此基础上完成你的代码，推送到自己的仓库可能遇到如下问题： error: remote origin already exists.表示远程仓库已存在。 因此你要进行以下操作： 1、先输入git remote rm origin 删除关联的origin的远程库 2、关联自己的仓库 git remote add origin https://gitee.com/xxxxxx.git 3、最后git push origin master，这样就推送到自己的仓库了。
